## 🟡 Important

### Q3 🟡 `autoMode` persists across runs (stranded tier)

**Where**
`state.ts:76-80` · `watchdog.ts onWorkflowEnd` · `autonomy.ts:110`

**Code**
```ts
export function clearActiveRun(): void {
	const s = getState();
	s.activeRunId = undefined;
	s.resumingRunId = undefined;
}
```

**Why**
`clearActiveRun` (and watchdog's `onWorkflowEnd`, which only calls `clearResumeAttempts` + `clearActiveRun`) never touches `autoMode`. Once a user sets `safe`/`unattended`, the tier stays live for every subsequent run in the same Pi process — `directiveFor(getAutoMode())` at `autonomy.ts:110` keeps injecting it, silently auto-crediting artifacts / auto-answering decisions in later runs the user never opted those runs into. This is precisely the autonomy-scope-creep failure mode that `258b8d7` was rushed out to fix 12 min after `b1164c3`. README documents "in-memory, lost on restart" but not "persists across runs within a process."

**Fix**
Reset `autoMode` to `"off"` in `clearActiveRun` (or in `onWorkflowEnd`) so each run starts at the explicit default; require re-opt-in per run.

**Alt**
Keep the tier sticky but emit a per-run notice ("full-auto `<tier>` still active") so the persistence is visible rather than silent.

### Q1 🟡 Unguarded `readFileSync` aborts resume instead of cold re-run

**Where**
`commands.ts:101`

**Code**
```ts
const data = parseFrontmatter(readFileSync(found.abs, "utf8"));
```

**Why**
The read sits inside `resumeCmd`'s outer try/catch but has no inner guard. A locked/deleted/unreadable artifact throws, the outer catch notifies an error and returns, and `await rt.resumeWorkflowByRunId(...)` is never reached. The block comment and README ("appendStage failure falls through to a cold re-run (safe degradation)") cover only `appendStage` returning `false`, not a `readFileSync` throw — so the documented safe-degradation contract is not honored, and under the 429 loop each tick re-enters the same throwing path until the 8h cap.

**Fix**
Wrap `readFileSync`/`parseFrontmatter` in a try/catch that, on failure, falls through to the plain `resumeWorkflowByRunId` cold re-run (matching the documented contract).

### Q4 🟡 Imminent reset wraps to +24h → premature give-up

**Where**
`ratelimit.ts:119-122`

**Code**
```ts
const targetSod = hour * 3600 + min * 60;
const nowSod = cur.h * 3600 + cur.m * 60 + cur.s;
let deltaSec = targetSod - nowSod;
if (deltaSec <= 0) deltaSec += 24 * 3600; // already passed today → next day
```

**Why**
`targetSod` is minute-granular while `nowSod` includes seconds. When the current time is N seconds into the target minute (reset imminent), `deltaSec = -N ≤ 0` wraps to ~86370 s. `armRetry` clamps with `Math.min(86370000, remaining≈8h) = 8h`; the timer fires at the wall-clock deadline where `remaining <= 0`, so the loop gives up instead of retrying the reset that was seconds away.

**Fix**
Floor `nowSod` to minute granularity (drop `cur.s`) before the subtraction, or treat a small negative delta (`> -60s`) as "now" rather than wrapping +24h.

### Q2 🟡 Substring stage match can credit the wrong artifact

**Where**
`continue.ts:108` (called from `commands.ts:99`)

**Code**
```ts
if (c.rel.toLowerCase().includes(lower)) return c;
```

**Why**
`findNewestArtifactMd(cwd, last.stage)` matches the stage name as a free substring of the artifact path, not a path segment. A short/common stage name (e.g. `design`) matches unrelated paths (`.../redesign/...`, `.../by-design/...`), so the newest such artifact is credited and the workflow advances on the wrong stage's output — partially defeating the I1/Q1 stage-scope fix this branch was added for.

**Fix**
Match on a path-segment boundary (split on `/` and compare the stage directory segment, or anchor the substring with separators) before falling back to the frontmatter `topic`/`stage` scan.

### I1 🟡 Force-clear of the shared resuming guard can drive duplicate resumes − *(demoted from 🔴; duplicate-credit not reproducible)*

**Where**
`ratelimit.ts:177-180` · `watchdog.ts session_start`

**Code**
```ts
if (isResuming(runId)) clearResuming(runId);
markResuming(runId); // suppress the session_start storm the resume's own spawns trigger (same guard the watchdog uses)
safeNotify(ctx, `rpiv-wfex: retrying @${runId} after usage limit…`, "info");
void Promise.resolve(pi.sendUserMessage(`/wfex resume @${runId}`)).catch((err) => {
```

**Why**
`watchdog` *bails* when `isResuming(runId)` is set; `ratelimit` instead force-clears that shared guard and re-marks it for its own resume. If a watchdog resume is genuinely in flight for the same run, the force-clear discards the watchdog's interlock marker and a second `/wfex resume` is queued — two resumes for one run, with no idempotency key on the message. **Verification weakened the original 🔴:** the credit block (`commands.ts:97-102`, `readLastStage`→`appendStage`) contains no `await`, so the read-check-write is atomic in single-threaded JS — once one resume appends the `completed` row, the other reads `completed` and skips. A *two-completed-row* trace is therefore not reproducible; the residual risk is duplicate **resume turns** (and the cold re-runs / side effects they carry), not duplicate credit rows.

**Fix**
Have `ratelimit` follow the watchdog's discipline — skip the tick when `isResuming(runId)` is already set by another resumer — rather than force-clobbering the guard; or scope `RETRY_SLOT`'s mark to detect a watchdog-owned guard.

## 🔵 Suggestions

### S1 🔵 `safe` tier auto-credits like `unattended` (documented, not silent) − *(demoted from 🟡)*

**Where**
`commands.ts:96` vs `autonomy.ts:93-94`

**Fix**
Consider gating the auto-credit behind a confirmation in `safe` (distinct from `unattended`) so the directive-level promise "safe still asks on substantive decisions" extends to the resume credit path. Note: README.md:96-99 already documents that the credit fires for both `safe` and `unattended`, so this is a design choice to revisit, not a hidden contradiction — surfaced for triage.

### Q5 🔵 HTTP-date `retry-after` not parsed → silent polling fallback

**Where**
`ratelimit.ts:218`

**Fix**
Accept an HTTP-date `retry-after` in `parseResetDelayMs` (parse to a Date, diff from now) in addition to the integer-seconds and `resets HH:MM (TZ)` forms; otherwise a spec-legal header silently degrades to 10-min polling.

---

### Q6 🔵 No `onWorkflowEnd` hook — retry slot self-clears late

**Where**
`ratelimit.ts` (`registerRateLimitRetry`)

**Fix**
Register an `onWorkflowEnd` lifecycle handler (as watchdog does) that calls `clearRetry()` on clean run-end, instead of relying on the next `fireRetry` tick's `getActiveRunId() !== runId` self-clear up to a poll interval later. Best-effort today (timer is `unref`'d), but leaves stranded slot state across runs.

---

### Q7 🔵 Retry loop bounded only by wall-clock, no attempt cap

**Where**
`ratelimit.ts:152-158`

**Fix**
Consider a per-run attempt counter alongside `MAX_RETRY_WALL_MS` (mirroring watchdog's `MAX_RESUME_ATTEMPTS=3`) to bound cold-re-run count of non-idempotent stages, not just elapsed time. Note: README.md:92 documents the "Bounded retry ≠ idempotent stages" caveat, so this is a known, accepted gap.

---

### G1 🔵 Retry timer state machine + credit block untested

**Where**
`ratelimit.selfcheck.ts:55-61`

**Fix**
The selfcheck drives the state machine by writing the private `Symbol.for("@flex/rpiv-wfex:ratelimit")` slot directly and never invokes `armRetry`/`fireRetry`, and the `commands.ts` credit block has no test. Add a real-invocation test (fake `pi`/`ctx`, fake timers) covering the wall-cap give-up path and the credit→appendStage path.
